Automated Queue Monitoring System

AI-powered computer vision system that detects, tracks, and counts people in queues, filters out passers-by using temporal analysis, estimates wait times, and streams results to a real-time web dashboard.

Section 1

Project Overview

This project solves the problem of manually monitoring queues in places like banks, airports, hospitals, and retail stores. Instead of relying on human observation, the system uses a camera feed and artificial intelligence to automatically count how many people are standing in line, ignore people who are just walking past, and predict how long each person will need to wait.

Key Capabilities

  • Person Detection — Uses YOLOv8 to identify every person in each video frame
  • Persistent Tracking — Assigns a unique ID to each person and follows them across frames
  • ROI Filtering — Only counts people inside a user-defined queue area
  • Temporal Filtering (3-second rule) — Eliminates false positives from passers-by
  • Wait Time Estimation — Predicts remaining wait time per person
  • Real-Time Web Dashboard — Live video feed + count + avg wait time in a browser
  • Baseline Comparisons — Academic evaluation against two alternative methods

End-to-End Data Flow

Video Frame → CLAHE Preprocessing → YOLOv8 Detection → Centroid Tracking → ROI Check → 3s Temporal Filter → Wait Time Estimation → Web Dashboard

Section 3

Object Detection — YOLOv8 Nano

YOLO (You Only Look Once) is a family of real-time object detection models. Unlike two-stage detectors (like Faster R-CNN) that first propose regions then classify them, YOLO is a single-stage detector — it processes the entire image in one pass, making it much faster.

Why YOLOv8 Nano?

Model Selection Rationale

YOLOv8n (nano) is the smallest variant with only 3.2 million parameters. It was chosen because:

  • Runs in real-time even on CPU (no dedicated GPU required)
  • Pre-trained on the COCO dataset (80 object classes including "person")
  • Sufficient accuracy for detecting people at typical queue camera distances
  • Low memory footprint — suitable for deployment on edge devices

Configuration Parameters

ParameterValueExplanation
imgsz640Input image resized to 640×640 pixels before inference
conf0.35Minimum confidence threshold — detections below 35% are discarded
classes[0]Class 0 = "person" in COCO — all other 79 classes are ignored

Section 4

Image Preprocessing — CLAHE

CLAHE stands for Contrast Limited Adaptive Histogram Equalization. It is applied to every frame before passing it to YOLOv8 to improve detection in poor lighting conditions.

Why Preprocessing Is Needed

Real-world queue videos often suffer from uneven lighting — shadows under awnings, bright sunlight on one side, dim indoor lighting. Without preprocessing, YOLOv8 may miss people in dark areas or produce false positives in overexposed regions.

Step-by-Step Process

def apply_clahe_bgr(img):
    # Step 1: Convert BGR → LAB color space
    #   L = Lightness, A = Green-Red, B = Blue-Yellow
    lab = cv2.cvtColor(img, cv2.COLOR_BGR2LAB)

    # Step 2: Split into individual channels
    l, a, b = cv2.split(lab)

    # Step 3: Apply CLAHE to the Lightness channel only
    #   clipLimit=3.0 prevents over-amplification of noise
    #   tileGridSize=(8,8) divides image into 64 local regions
    clahe = cv2.createCLAHE(clipLimit=3.0, tileGridSize=(8,8))
    l2 = clahe.apply(l)

    # Step 4: Merge back and convert to BGR
    merged = cv2.merge((l2, a, b))
    return cv2.cvtColor(merged, cv2.COLOR_LAB2BGR)

Key Design Decisions

  • LAB color space — By working in LAB, we enhance brightness (L channel) without distorting the actual colors (A, B channels)
  • clipLimit = 3.0 — Limits contrast amplification to prevent noise from being exaggerated
  • tileGridSize = (8, 8) — The image is divided into 64 tiles, each getting its own histogram equalization. This makes the enhancement adaptive — different areas get different adjustments
Section 5

Object Tracking — Centroid Tracker

Problem: YOLOv8 gives us bounding boxes per frame, but no identity. It cannot tell us whether "person A in frame 10" is the same individual as "person A in frame 11." Without tracking, we can't count unique people or measure how long someone has been in the queue.

Solution: A custom Centroid Tracker that maintains unique IDs across frames using Euclidean distance matching.

How It Works — Step by Step

Step 1: Compute Centroids

For each bounding box [x1, y1, x2, y2], calculate the center point: centroid = ((x1+x2)/2, (y1+y2)/2)

Step 2: Build Distance Matrix

Compute the Euclidean distance between every existing tracked centroid and every newly detected centroid. This produces an N×M matrix where N = existing objects and M = new detections.

Step 3: Greedy Matching

Sort rows by minimum distance. For each existing object, find the closest new detection. If the distance is within max_distance (80 pixels), they are matched — the existing object's position is updated to the new centroid.

Step 4: Handle Disappearances

If an existing object isn't matched to any new detection, its disappeared counter increments. After max_disappeared (15) consecutive frames without a match, the object is deregistered — the person is assumed to have left.

Step 5: Register New Objects

Any new detection not matched to an existing object is registered as a new person with a fresh unique ID.

Key Parameters

ParameterValueMeaning
max_disappeared15 framesNumber of consecutive frames a person can be undetected before removal
max_distance80 pixelsMaximum centroid movement between frames to be considered the same person

Section 6

Region of Interest (ROI)

Not every person visible in the camera is standing in the queue. The ROI is a user-defined polygon that marks the exact area where the queue exists. Only people whose centroid falls inside this polygon are considered for counting.

Interactive ROI Selection

When the system runs for the first time, an OpenCV popup window appears showing a frame from the video. The user clicks to place polygon vertices around the queue area. The coordinates are saved to roi_points.json and automatically reloaded on subsequent runs.

Controls

  • Left Click — Add a polygon vertex (numbered on screen)
  • D key — Undo the last placed vertex
  • Enter / Q — Finish selection and save

Point-in-Polygon Test

def inside_roi(x, y, polygon):
    # Returns True if point (x,y) is inside the polygon
    # Uses OpenCV's pointPolygonTest — returns ≥0 if inside or on edge
    return cv2.pointPolygonTest(polygon, (int(x), int(y)), False) >= 0

This function is called for every tracked person on every processed frame. Only centroids that return True contribute to the queue count.

Section 7

Temporal Filtering — The 3-Second Rule

This is the core innovation of the proposed method. A person is only counted as "in queue" if they remain continuously inside the ROI for at least 3 consecutive seconds.

Why This Matters

Without temporal filtering, someone walking past the queue who briefly enters the ROI would be counted as a queue member. The 3-second rule eliminates these false positives because a passer-by will typically cross the ROI in under 2 seconds.

How the Frame Threshold Is Calculated

# Video FPS ≈ 60, SKIP_FRAMES = 4
# We process every 4th frame → 15 processed frames per second
# 3 seconds × 60 FPS ÷ 4 skip = 45 processed frames needed

frames_required = max(1, int(ceil(3.0 * fps / SKIP_FRAMES)))

Consecutive Sequence Check

The system doesn't just count total appearances — it checks for an unbroken consecutive sequence. If a person is detected in the ROI on frames 10, 11, 12, 13, ... 54, 55 (45 consecutive processed frames), they become "confirmed."

# Check if the last N entries in the deque are consecutive
seq_len = 1
for i in range(len(dq) - 1, 0, -1):
    if dq[i] - dq[i-1] <= 1:
        seq_len += 1
    else:
        break

if seq_len >= frames_required:
    ids_confirmed.add(oid)  # Person is now counted

Visual Indicator

  • Red bounding box — Person detected but NOT yet confirmed (under observation)
  • Green bounding box — Person confirmed (passed the 3-second rule, counted in queue)

The "In Queue" count shown on screen only includes green (confirmed) individuals currently inside the ROI.

Section 9

Wait Time Estimation

The system predicts how much longer each person in the queue will need to wait. This is based on observing how long previous people spent in the queue before leaving.

Algorithm

Step 1: Track Entry Times

When a confirmed person first appears inside the ROI, record the current video timestamp as their entry_time.

Step 2: Detect Departures

Each frame, compare the set of confirmed IDs currently in the ROI against the previous frame's set. Any ID present before but missing now has departed.

Step 3: Compute Queue Duration

For each departed person: queue_duration = departure_time - entry_time. This is added to a running list of all departed durations.

Step 4: Average Service Time

After at least 3 departures (to avoid noisy estimates), compute: avg_service_time = sum(all_durations) / count(all_durations)

Step 5: Per-Person Prediction

For each person currently in the queue: estimated_remaining = max(0, avg_service_time - time_already_spent)

# Departure detection
departed = prev_confirmed_in_roi - current_confirmed_in_roi
for oid in departed:
    if oid in entry_time:
        departed_durations.append(current_video_time - entry_time[oid])

# Average (only after 3+ departures)
if len(departed_durations) >= 3:
    avg_service = sum(departed_durations) / len(departed_durations)

# Per-person estimate
remaining = max(0, avg_service - elapsed)

The estimated wait is shown as ~Xm Ys below each person's bounding box on the video, and the average is displayed on the web dashboard.

Section 11

Real-Time Web Interface

The web dashboard allows monitoring the queue from any browser on the local network, without needing to run OpenCV windows.

Architecture

Backend — Flask + Threading

  • The detection pipeline runs in a background daemon thread, independent of Flask's request handling
  • Two shared resources protected by thread locks:
    • _state — dictionary with count, avg_wait, status (guarded by _state_lock)
    • _latest_frame — the most recent annotated frame (guarded by _frame_lock)

HTTP Endpoints

EndpointMethodFunction
GET /GETServes the HTML dashboard page
POST /startPOSTLaunches the detection thread; returns error if already running
GET /video_feedGETMJPEG stream — continuously sends JPEG frames as multipart HTTP response
GET /streamGETServer-Sent Events — pushes JSON updates (count, avg_wait, status) every 300ms
GET /statusGETReturns current state as JSON (for polling)

MJPEG Video Streaming

The /video_feed endpoint uses the MJPEG (Motion JPEG) protocol. Each annotated frame is encoded as a JPEG image and sent as part of a continuous multipart HTTP response. The browser's <img> tag with this URL auto-refreshes to show each new frame — no JavaScript needed for the video itself.

# MJPEG stream generator
def generate():
    while True:
        with _frame_lock:
            frame = _latest_frame
        if frame is not None:
            _, buf = cv2.imencode(".jpg", frame, [cv2.IMWRITE_JPEG_QUALITY, 70])
            yield (b"--frame\r\n"
                   b"Content-Type: image/jpeg\r\n\r\n" + buf.tobytes() + b"\r\n")
        time.sleep(0.05)

Server-Sent Events (SSE)

SSE is a lightweight one-way communication channel from server to browser. Unlike WebSockets, it uses standard HTTP and is simpler to implement. The browser connects once via EventSource, and the server pushes data whenever it changes.

// Browser-side JavaScript
const evtSource = new EventSource('/stream');
evtSource.onmessage = function(event) {
    const d = JSON.parse(event.data);
    // d.count, d.avg_wait, d.status are now available
    updateUI(d);
};

Frame-Rate Throttling

Without throttling, the detection loop would process frames as fast as possible, making the video appear to play at 5-10x speed. A time.sleep() at the end of each iteration ensures real-time playback:

target_delay = SKIP_FRAMES / fps  # 4/60 ≈ 0.067 seconds
elapsed = time.time() - loop_start
if target_delay - elapsed > 0:
    time.sleep(target_delay - elapsed)

Baseline Comparisons

To scientifically validate the proposed method, two baseline approaches are implemented and compared.

MethodDetectionTrackingTemporal FilterPurpose
Proposed YOLOv8n Centroid Tracker 3 seconds The main method being evaluated
Baseline 1 YOLOv8n Centroid Tracker None Proves temporal filtering reduces false positives
Baseline 2 YOLOv8n DeepSORT 3 seconds Proves centroid tracker is competitive with a heavier tracker

DeepSORT — How It Differs

DeepSORT (Deep Simple Online and Realtime Tracking) uses a MobileNet CNN to extract visual appearance features from each detected person. It then combines:

  • Appearance similarity (cosine distance between CNN embeddings) — recognizes people by how they look
  • Motion prediction (Kalman filter) — predicts where a person will be in the next frame
  • Hungarian algorithm — optimal assignment of detections to tracks

While more robust in theory, DeepSORT is significantly more computationally expensive due to the CNN embedding extraction. For queue monitoring where people move slowly and occlusion is limited, the simpler centroid tracker performs comparably.

Evaluation Metrics

The system is evaluated against manually annotated ground truth data. A human reviewer counts the actual number of people in each saved frame, and these counts are compared to the model's predictions.

MetricFormulaWhat It Measures
MAE mean(|GT - Pred|) Average absolute error per frame — how far off the count is on average
RMSE sqrt(mean((GT - Pred)²)) Same as MAE but penalizes large errors more heavily
Precision TP / (TP + FP) Of all people the model counted, what fraction actually existed
Recall TP / (TP + FN) Of all real people, what fraction did the model detect
F1 Score 2 × P × R / (P + R) Harmonic mean of precision and recall — balanced measure
Queue Accuracy 1 - (total_error / total_GT) Overall accuracy as a percentage — the headline metric
